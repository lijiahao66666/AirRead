# AirRead 代码审计与瘦身任务

目标：在不改变现有功能的前提下，删除无用文件/无用代码，增强关键路径健壮性，并降低维护成本。

## 产出
- 一份“可删除/可合并/可重构”的清单（含文件路径、原因、风险评估）
- 一份“必须补齐的健壮性/边界处理”清单（含建议改法）
- 一轮实际改动：删除低风险无用文件与明显死代码、修复高价值健壮性问题，并保持构建与测试通过

## 审计顺序（建议按优先级）
1) 构建与发布链路
- iOS：Debug/Release 构建，真机/模拟器运行，插件与权限配置核查
- Android：Debug/Release 构建，ABI/so 裁剪核查，启动/权限/崩溃日志核查
- Flutter：`flutter test`、`flutter analyze`（区分 error/warn/info）

2) 本地推理（Local LLM）专项
- 模型下载：断点续传/失败重试/文件校验/磁盘空间不足处理
- 推理链路：初始化失败提示、流式取消、异常收敛（不让 UI 卡死）
- iOS/Android 差异：Prompt 模板、stop 行为、token 上限策略一致性

3) 阅读器（Reader）专项
- 章节解析（EPUB/TXT）：标题/封面/目录/前言识别与跳过策略
- 分页：极端宽高、字体/行高、空白/特殊字符、首章杂项内容（图片/换行）健壮性
- 选择/复制/翻译：跨页高亮、段落切分一致性

4) 代码瘦身与清理
- 删除未引用 Dart 文件/类/方法（以实际引用与运行路径为准）
- 清理临时调试输出（保留必要的错误日志）
- 统一错误处理：避免吞异常、避免无意义的 fallback 值

## 验证标准
- `flutter test` 全通过
- iOS：`xcodebuild`（sim/device）可构建；真机安装可启动并可用本地模型（若设备支持）
- Android：`gradlew assembleDebug`/`assembleRelease` 至少 Debug 可构建
- 关键路径手测：导入一本书 → 进入阅读 → 翻译/问答 → 本地模型流式输出
